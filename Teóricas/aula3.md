# Modelos Conceituais

## 1. What is Conceptual Model?

- Define o que as pessoas podem fazer com o sistema, fornecendo uma visão clara das possibilidades de interação;
- Ele especifica os conceitos que a equipa precisa compreender para utilizar o sistema de forma eficiente;
- O modelo refere-se a `objetos`, `atributos` e `ações` dentro do dominio da tarefa, estabelecendo-se os mapeamentos necessários entre esses conceitos;
- Ele permite que a equipa organize as ideias antes de iniciar o desenho da interface do utilizador;
- Os principais componentes de um modelo conceitual são:
  - Os dados que os utilizadores podem manipular no sistema;
  - A organização dos dados dentro do sistema;
  - A ações que os utilizadores podem realizar com esses dados.

### 1. What is not Conceptual Model?
- O modelo conceitual não é a interface de utilizador e não descreve diretamente como o sistemas será visualmente apresentado;
- Ele não se refere à arquitetura técnica do sistema, nem aborda a aparência ou elementos gráficos como ações de mouse, teclas de atalho, etc;
- Além disso, o modelo concetual não especifica os tipos de intereção permitidos e não reflete o modelo mental do utilizador;

### 2.Modelo Mental:
- O Modelo mental representa a visão do utilizador sobre o sistema, ou seja, como ele compreende o funcionamento e as capacidades do sistema;
- O Modelo Conceptual serve como um intermediário entre o modelo de implementação, que reflete os aspetos técnicos do sistema, e o modelo mental do utilizador, que reflete a experiência do utilizador. Ele deve ajudar a criar representações que correspondam à visão e expectativas do utilizador.

## 2. Requirements and Task Selection
### 1. Functional Requirements
- Os `functional requirements` são um conjunto de ações que o sistema deve permitir, com base nos cenários e nas personas definidas;
- Estes requisitos devem incluir funcionalidades que satisfaçam as necessidades dos utilizadores de maneira especifica; ex:Login, gestão de fotos de perfil, etc. 

### 2. Task Selection
- `Task Selection` consiste em identificar atividades representativas que ilustram o uso do sistema. Essas tarefas devem cobrir os requisitos definidos e servir como base para futuras avaliações do sistema.
- As `Task Selection` devem ser reais e representativas, sem mencionar diretamente como a interface será utilizada. Elas devem ser específicas em termos de intenções e objetivos, misturando

### 3.Usability Requirements
- Os `Usability Requirements` são usados para avaliar a facilidade de uso do sistema, considerando a eficácia, eficiência e a satisfação dos utilizadores;
- A eficácia refere-se à qualidade com que o utilziador alcança os objetivos no sistema;
- A eficiência está relacionada aos recursos gastos pelo utilizador para atingir esses objetivos;
- Por fim, a satisfação mede a reação subjetiva dos utilizadores, em relação à experiência proporcionada pelo sistema;

## 3. Ideation
- `Ideation`(inovação) não começa com uma ideia; ideias surgem ao entender as necessidades do cliente e iterar no processo;
- `Qualidade vs Quantidade`: Trabalhos de maior qualidade vêm da experimentação contínua e aprendizado com erros;
- `Design Sprints`: Métodos ágil para gerar e testar ideias rapidamente;
- `Crazy 8`: Técnica onde cada elemento cria 8 ideias em 8 minutos, incentivando variedade e inovação.

## Sobre os Slides

1. **Qual das seguintes opções NÃO é uma característica de um modelo conceptual?**  
    1. Descreve o que as pessoas podem fazer com o sistema.  
    2. Define a arquitetura do sistema e os seus componentes.  
    3. Utiliza conceitos familiares aos utilizadores para facilitar a compreensão.  
    4. Ajuda a equipa a organizar ideias antes de desenhar a interface.  

2. **Qual é a principal diferença entre um modelo conceptual e um modelo mental do utilizador?**  
    1. O modelo conceptual é criado pelos *designers*, enquanto o modelo mental é formado pelos utilizadores.  
    2. O modelo conceptual foca-se na aparência da interface, enquanto o modelo mental aborda a funcionalidade.  
    3. O modelo conceptual é estático, enquanto o modelo mental evolui com a experiência do utilizador.  
    4. O modelo conceptual é complexo e detalhado, enquanto o modelo mental é simples e intuitivo.  

3. **No exemplo de John e as suas fotos, qual dos seguintes elementos é considerado um atributo do objeto "Foto"?**  
    1. Inserir foto  
    2. Data da foto  
    3. Criar evento  
    4. Partilhar arquivo  

4. **Porque é importante considerar metáforas familiares aos utilizadores ao criar um modelo conceptual?**  
    1. Para tornar o sistema mais visualmente apelativo.  
    2. Para simplificar o desenvolvimento do sistema pelos programadores.  
    3. Para reduzir o tempo de aprendizagem e facilitar a utilização do sistema.  
    4. Para eliminar a necessidade de realizar testes de usabilidade com utilizadores.  

5. **Qual das seguintes opções NÃO é um requisito funcional típico de uma aplicação móvel para gestão de finanças pessoais?**  
    1. Registar receitas e despesas.  
    2. Categorizar transações financeiras.  
    3. Gerar relatórios de despesas por categoria.  
    4. Escolher a cor de fundo da interface.  

6. **Ao definir as tarefas dos utilizadores para um sistema, qual das seguintes características é crucial?**  
    1. Devem ser especificadas em termos de como o utilizador irá interagir com a interface.  
    2. Devem ser complexas e desafiadoras para garantir que o sistema é robusto.  
    3. Devem ser realistas e representativas do uso real do sistema.  
    4. Devem focar-se nos aspetos técnicos da implementação do sistema.  

7. **Qual das seguintes métricas NÃO é normalmente utilizada para avaliar a eficiência de um sistema?**  
    1. Tempo médio para completar uma tarefa.  
    2. Número de cliques necessários para realizar uma ação.  
    3. Taxa de erros cometidos pelos utilizadores durante a utilização.  
    4. Nível de satisfação dos utilizadores com a interface.  

8. **Qual é o objetivo da técnica "Crazy 8" no processo de ideação?**  
    1. Avaliar a usabilidade de um protótipo através de testes com utilizadores.  
    2. Gerar um grande número de ideias em um curto período de tempo.  
    3. Documentar o modelo conceptual de um sistema de forma detalhada.  
    4. Definir os requisitos funcionais e de usabilidade de um sistema.  

9. **De acordo com o conceito de "Fail Fast Philosophy", qual é a abordagem ideal para lidar com falhas no processo de design?**  
    1. Evitar falhas a todo o custo, realizando testes exaustivos antes de qualquer lançamento.  
    2. Identificar e corrigir falhas o mais cedo possível no processo de design.  
    3. Esconder as falhas dos utilizadores para garantir uma experiência positiva.  
    4. Ignorar as falhas, pois elas são inevitáveis em qualquer sistema complexo.  

10. **Qual das seguintes afirmações sobre a importância da pesquisa com o utilizador é VERDADEIRA?**  
    1. A pesquisa com utilizador é dispendiosa e deve ser evitada para reduzir custos de desenvolvimento.  
    2. Os *designers* têm um conhecimento intuitivo das necessidades dos utilizadores e não precisam de realizar pesquisas.  
    3. A pesquisa com o utilizador é essencial para criar sistemas eficazes, eficientes e satisfatórios para os utilizadores.  
    4. A pesquisa com utilizador só é necessária após o lançamento de um produto para recolher *feedback* dos utilizadores.
